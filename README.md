# DND5R 职业/子职节奏榜

## 仓库内容

- `.codex`：项目级 Codex skills。这里保存评分流程、资料核查规则、文件结构、命名规则和 changelog 规则。
- `tools`：批量生成、校准或整理评分文件的本地脚本。这些脚本主要服务于已有工作流，不保证都是通用 CLI。
- `Rankings`：官方、UA、合作方与第三方公开内容的评分与评审产物。
- `DND5e_chm`：资料 submodule，指向 `https://github.com/DND5eChm/DND5e_chm`。用于核查最新 5.5e / 2024 内容，应保持更新。
- `DND2014_legacy`：资料 submodule，同样指向 `https://github.com/DND5eChm/DND5e_chm.git`，但固定在当前 revision。用于保留纯 5e 时代资料，尤其是未来复核 5e UA 子职；不要为了 5e UA 评审而更新它。

`homebrews` 目录不纳入本仓库。该目录目前属于个人创作与迭代区，不作为公开评分产物发布；与 homebrew 评审相关的流程规则仍保留在 `.codex` skills 中，方便本地使用。

## 初始化

克隆仓库后拉取 submodule：

```powershell
git clone --recurse-submodules git@github.com:royale011/DND5R_Rankings.git
```

如果已经普通克隆：

```powershell
git submodule update --init --recursive
```

更新最新资料时，只更新 `DND5e_chm`：

```powershell
git -C DND5e_chm pull
git add DND5e_chm
git commit -m "Update DND5e_chm submodule"
```

不要随手更新 `DND2014_legacy`。它的用途是保留当前 5e 时代资料快照。

## 如何运行官方评分

官方、UA、合作方、第三方公开内容的评分输出写入 `Rankings`。

推荐流程：

1. 使用 `.codex/skills/official-dnd-ranking-review/SKILL.md` 作为工作规则。
2. 先读 `Rankings/changelog.md`，确认当前评分标准和最近校准。
3. 用 `DND5e_chm` 核查职业、子职、专长、法术、物品、规则原文和中文译名。
4. 如需社区意见，必须确认评论讨论的是同一版本规则；不匹配的旧版评论只能作为背景，不可直接套用。
5. 修改对应的 `Rankings/[职业]/README.md`、子职文件或 `Rankings/构筑/*.md`。
6. 如果新增或修改非 homebrew 牧师领域评审，同时检查是否需要新增或更新 `Rankings/法师/神圣奇术/[牧师领域].md`。神圣奇术本体是 5E UA 法师子职，因此这些映射在法师与根 README 的跨来源排行榜中显示为 `神圣奇术（5E UA） - [牧师领域]`。
7. 如果更新了标准、校准逻辑或大范围结论，把记录添加到 `Rankings/changelog.md` 文件开头。

## 官方职业 / 子职评审标准

官方评审面向 `Rankings` 中的官方、UA、合作方、第三方公开内容。评分不是印象榜，也不是只看单一 DPR 的强度表；每个结论都应回到本地规则文本、版本匹配的社区讨论、职业底盘、子职增量、资源与行动经济。

核心标准：

- 资料必须先用 `DND5e_chm` 核查。职业、子职、特性、法术、专长、物品、怪物、规则术语和中文译名都不能靠记忆补全；社区评论也必须确认讨论的是同一版规则。
- 强度、功能与综合评级顺序固定为 `S+ > S > S- > A+ > A > A- > B+ > B > C > D > E > E- > F`。不新增 `SS`、`B-`、`C+`、`C-`、`D+`、`D-`、`E+` 等强度分档；`B+` 专门表示明显高于普通可玩 `B`、但仍不够 `A-` 的选项。设计评分额外允许 `B-`，只用于表达“低于普通可接受 B、但还没差到 C”的设计问题，不参与排行榜排序。
- `综合评分` 不是分项平均。职业文件评价的是职业底盘在该阶段的实战价值；子职文件评价的是“职业底盘 + 子职增量”形成的完整角色选项。理由中必须说明底盘贡献、子职真正增加了什么，以及限制器、机会成本或桌面依赖。
- `S+` 只给全生态级别的破局顶点，例如低成本重塑行动经济、豁免 / d20 结果、专注归属、队伍防御或法术投送的选项。只是本职业最强，通常不够 `S+`。
- 满施法者和近满施法者要防止“底盘压缩”。高阶满施法职业本身很强，但普通有用子职通常应落在 `S-` 或 `A+`；只有子职实质改善胜利条件、法术成功率、反魔法、队伍保护、专注稳定、法术准备 / 法术书经济或行动经济时，才应上探 `S` 或 `S+`。
- 非满施法者不能按满施法标准扣分，也不能只因伤害高就上调。评审要同时看目标接触、飞行 / 隐形 / 地形处理、命中稳定、爆发与续航、动作 / 附赠动作 / 反应压力、武器精通、控制附带效果、防御恢复、团队杠杆、技能与探索、物品依赖和高阶扩展。
- 半施法、三分之一施法和 gish 要把法术放进行动经济里评价。延迟法术环、准备 / 已知法术压力、专注冲突、附赠动作标记、打击、伙伴指令和攻击动作之间的冲突都要算入结论。
- 规则互动要看真实资源：法术位、免费施法、短休 / 长休、专注、材料费、卷轴可用性、法术书、召唤 / 伙伴指令、职业资源和不利中断后果。旧版附赠动作施法记忆不能替代 2024 `每回合仅能施展一道消耗法术位的法术` 的实际文本。
- `武器精通`、`起源专长`、常见专长、物种、普通优化和常规魔法物品可以进入评价，但不能默认稀有 / 传奇物品、理想停机时间或 DM 特供奖励。施法者和武器角色的外部资源假设要对称。
- 阶段内时间点必须写清。10、11、14、15、17、18、20 级特性如果只覆盖阶段后半段，`综合评分` 理由要说明是全阶段表现、后段尖峰，还是阶段内分段。
- 设计评分独立于强度评分。`设计质量`、`主题`、`能力设计质量`、`主题与能力关联度` 要评价边界、清晰度、版本兼容、主题表达和玩法结构；高强度不自动等于好设计。

评审文字应具体到特性与规则，而不是重复分项表。例如：一个 14 级才取得核心现实改写能力的子职，Tier 3 理由应说明 11-13 级和 14-16 级的差异；一个依赖优势、标记或专注的武器子职，应先检查同职业底盘、武器精通、法术、专长和物品是否已经提供可替代入口。

可用脚本在 `tools` 下。常用示例：

```powershell
python .\tools\rebuild_class_leaderboards_from_reviews.py
python .\tools\update_root_summary.py
```

其他 `rewrite_*` 脚本多数是针对历史某一轮重写或校准的专用脚本。运行前先读脚本内容，确认它的目标文件和假设。

## 如何添加 Homebrew 内容

本仓库不会提交 `homebrews`。如果本地需要评审 homebrew，目录建议保持为：

```text
homebrews/
  [原始 homebrew txt / md / pdf 等]
  Rankings/
    changelog.md
    [职业或类型]/
      [子职名].md
      Archive/
    法师/
      神圣奇术/
        [映射领域名].md
        Archive/
```

新增 homebrew 原文后，使用 `.codex/skills/homebrew-dnd-ranking-review/SKILL.md` 作为工作规则。评审时仍要参考：

- `Rankings/changelog.md`
- 已有 `Rankings` 官方/合作方/第三方评分文件
- `DND5e_chm` 中的官方规则文本
- `homebrews/Rankings/changelog.md` 中的 homebrew 专用标准

Homebrew 文件如果同时有英文和中文，以中文作为 `原文` 归档内容，除非明确说明相反。

## 如何运行 Homebrew 评分

Homebrew 评分输出写入 `homebrews/Rankings`，不写入根 `Rankings`。

推荐流程：

1. 读取 `homebrews/Rankings/changelog.md` 和根 `Rankings/changelog.md`。
2. 读取当前 homebrew 原文。
3. 如果已有评审文件，比较当前中文原文和评审文件中的 `## 原文`。
4. 如果原文改变，先归档旧评审，再生成新评审。
5. 如果只是评分标准或文字表达改变，原文未变，可以直接原地更新，不必归档。
6. 核查 homebrew 提到的所有官方内容，不用记忆判断法术、专长、职业能力或怪物数据。
7. 在评审文件内写清 `修改建议`；不要维护 `homebrews/Rankings/README.md` 作为总建议文件。
8. Homebrew 牧师领域默认不映射到神圣奇术；只有明确要求时才在 `homebrews/Rankings/法师/神圣奇术` 创建 standalone 映射评审，且该目录不维护 `README.md`。

Homebrew 评审必须包含 `原文`，并在需要时包含：

- `设计相关评分 - 个人期望`
- `官方RAW修复建议`
- `修改建议 - 公开发布`
- `修改建议 - 个人意图`
- 改编来源核查，例如 `基于...改编`

## Homebrew 职业 / 子职评审标准

Homebrew 评审继承官方评审的通用强度标准，但目的不同：它不是公开仓库的一部分，而是为本地创作迭代保留版本、指出问题并给出可修改方向。评审输出仍要和官方、UA、合作方、第三方公开内容放在同一个强度生态中比较，不能因为是 homebrew 就降分或宽待。

Homebrew 专用标准：

- 输出只写入 `homebrews/Rankings`，根 `Rankings` 只作为校准基准。不要把私有 homebrew 评审、原文或示例写进公开 README。
- Homebrew 神圣奇术映射是显式请求才执行的特殊评审，不跟随官方流程自动生成；归档也放在 `homebrews/Rankings/法师/神圣奇术/Archive`。
- 每个评审必须包含 `原文`。如果原文件同时有中英文，以中文原文归档；如果中文原文相对已有评审发生变化，先按归档规则保存旧评审，再生成新评审。
- 评分时严格对照官方 / UA / 合作方 / 第三方公开内容。越界、过强、过弱或规则空洞都要进入平衡评分，而不是先假设作者会修正。
- 如果原文含 `附注`、`Note`、`设计意图`、`个人意图` 等内容，普通强度评分仍按公开生态标准；普通 `设计相关评分` 也只评价公开文本、机制、清晰度、主题表达和生态健康，不引用附注内容作为理由。另加 `设计相关评分 - 个人期望`，只在对应方面说明作者自己的私桌目标、强度目标、作者幻想或改编目的是否被达成。
- 如果原文写明 `基于...改编`，要核查来源材料，再评价它转换到 5.5e / 2024 规则后的保真度、必要改动和失真点。
- Homebrew 提到的官方职业、子职、法术、专长、物品、怪物、状态或规则，必须回到 `DND5e_chm` 核查。不要按旧版记忆评价，例如法术机制、施法时间、卷轴、专注和职业资源都可能已经变化。
- 如果 homebrew 使用了官方 2024 已有但 RAW 边界不清的设计模式，例如 `祈愿术` 复制法术或牧师 `进阶神圣干预` 选择 `祈愿术`，不要要求 homebrew 单独修补官方文字。评审应说明官方 RAW 本身存在桌裁敏感点，但这不是 homebrew 的 `必须修改` 项，也不因缺少额外监管句单独降低设计评分。
- 对这种官方 RAW 模糊相关的强度 / 功能 / 综合评分，先做横向校准：查明哪些官方职业、子职、法术、职业特性、构筑或法表拥有相同或相近的越界资源，记录其取得等级、资源频率、动作成本、法表 / 准备经济、当前评分，以及该资源在评分中承担的具体角色；再结合 homebrew 的底盘、到位等级、资源链、协同、冷却、范围和桌面负担决定实际档位。只有找不到可用官方参照或已评审类比时，才讨论 RAI / RAW 会怎样影响强度、功能或平衡评分。
- 如果评审需要给出“官方 RAW 应如何修”的方案，放入独立 `## 官方RAW修复建议` section，并在目录中列出。该 section 只讨论官方规则修复口径及其桌面影响，不写成 homebrew 必须修改项；评分依据仍优先来自横向校准。
- `修改建议` 写在每个评审文件里，不维护公开的 homebrew 总建议页。有个人设计目的时，拆成 `修改建议 - 公开发布` 和 `修改建议 - 个人意图`。
- 修改建议要区分：`必须修改`、`可以保留`、`强但可接受`、`仅公开发布时是问题`、`主观口味批评`。不要把“和官方子职有重叠”自动判为无效；只有当它在官方子职的主职能上明显压过原选项，才把重叠作为机械问题。
- `S+` 标准和官方相同：必须触及整个生态的顶点。只是超过本职业边界、超过普通 homebrew，或在私桌目标里很强，都不自动等于 `S+`。

公开发布视角和个人意图视角应分开写。例如，一个子职若给出无专注的强控制、额外行动或长期队伍增益，公开发布评价要比较同级官方法术和子职的专注、持续、目标、豁免、资源频率和可反制性；如果作者说明这是给高强度私桌使用，个人意图部分可以承认它达成目标，但不能改写普通强度评级。

## Homebrew 归档规则

当 homebrew 原文发生变化时，先把当前评审文件复制到：

```text
homebrews/Rankings/[职业或类型]/Archive/[子职名] - [YYYYMMDD HHMMSS].md
```

然后再生成新的评审文件。归档判断以“当前 homebrew 文件中的中文原文”和“评审文件 `## 原文` 区块”比较为准。

`homebrews/Rankings/changelog.md` 采用倒序记录：最新记录写在文件开头。

## 贡献评分更新

评分更新应遵守以下原则：

- 必须使用 `DND5e_chm` 核查当前规则文本和中文译名。
- 不要用模型记忆替代规则核查。
- 不要把 2014 特性误写到 2024 职业或子职中。
- `综合评分` 必须来自该文件的实际分项分析，不要只改排行榜。
- 如果更新排行榜，排行榜的等级和理由必须取自对应子职/构筑文件的 `综合评分` 表。
- 构筑目录只收录真正的多职业或跨系统构筑，不收录纯职业优化包。
- 官方、UA、合作方、第三方内容可以各自获得真实强度评分；但排行榜收录规则要按 skill 执行。
- 根 `Rankings/README.md` 的 `分阶段子职排行榜` 只放官方第一方、EGW 项目例外和有效构筑；UA、合作方、第三方子职放入独立排行榜。神圣奇术映射即使对应官方牧师领域，也按神圣奇术本体的 5E UA 来源，只进入根 `UA/合作方/第三方子职分阶段排行榜`。
- 如果改变评分标准或校准逻辑，在 `Rankings/changelog.md` 文件开头新增记录。

提交前建议检查：

```powershell
git status --short
git diff --check
```

确认没有提交本地私有内容：

```powershell
git check-ignore -v homebrews homebrews\Rankings
```

## 贡献 Skill 更新

Skill 是本仓库评分一致性的核心。修改 `.codex/skills/*/SKILL.md` 时：

- 写清触发场景、输出路径、命名规则和禁止事项。
- 同步更新根 `README.md` 中对应的使用说明。
- 如果变更会影响既有评分标准，在对应 changelog 文件开头记录。
- 官方评分规则改动记录到 `Rankings/changelog.md`。
- Homebrew 评分规则改动记录到 `homebrews/Rankings/changelog.md`，但该文件本身不提交到本仓库。
- 不要让官方与 homebrew 输出路径混用。

Skill 更新后，至少抽查一个已有评分文件，确认新规则不会和当前文件结构冲突。
